# 🧠 Document Intelligence RAG

Быстрый и точный корпоративный поисковик по документам на базе **Hybrid RAG** (векторный поиск + полнотекстовый поиск). Сервис позволяет загружать документы (PDF, Word, TXT, MD), индексировать их и задавать вопросы естественным языком с получением точных ответов и ссылок на источники.

## ✨ Ключевые особенности
- **LangGraph Pipeline:** запрос проходит через граф из трёх узлов — `retrieve → grade → generate`.
- **Гибридный поиск (Hybrid Retrieval):** комбинация семантического поиска (`NVIDIA E5-v5`) и лексического поиска по ключевым словам (`BM25`).
- **Reciprocal Rank Fusion (RRF):** умное слияние результатов обоих поисковых алгоритмов.
- **Grade-фильтрация:** нерелевантные чанки отсеиваются до отправки в LLM, снижая галлюцинации.
- **Параллельная индексация:** батчи эмбеддингов вычисляются конкурентно с ограничением через `Semaphore`.
- **Тип поиска в ответе:** поле `retrieval_type` (`hybrid` / `vector_only` / `bm25_only`) позволяет фронтенду показать, как был найден ответ.
- **Точность и Уверенность (Confidence Score):** откалиброванная оценка от 1% до 100%; при BM25-only фиксируется на 40%.
- **Чанкинг с перекрытием:** алгоритм дробления текста не разрывает смысл предложений.
- **Устойчивость к сбоям API:** автоматический retry с экспоненциальным backoff при таймаутах и ошибке 429.
- **Защита от гонки данных:** блокировка на уровне файла исключает дублирование при параллельной индексации.
- **Автоочистка очереди загрузки:** файлы, не проиндексированные в течение 1 часа, автоматически удаляются из памяти.

## 🛠 Технологический стек
* **Backend:** Python 3.10+, FastAPI, Uvicorn
* **RAG Pipeline:** LangGraph (граф retrieve → grade → generate)
* **Vector DB:** ChromaDB (локальная персистентная база)
* **Keyword Search:** rank-bm25 (в оперативной памяти)
* **LLM & Embeddings:** OpenAI Async Client (NVIDIA NIM endpoints)
* **Retry / Resilience:** tenacity (exponential backoff для API-вызовов)
* **Frontend:** Vanilla HTML5, CSS3, JavaScript (без тяжелых фреймворков)

## 💾 Перенос и резервное копирование базы данных

ChromaDB хранит все проиндексированные документы в обычной папке на диске (`chroma_data/` в корне проекта). Это значит:

- **Перенос на другой ПК:** Просто скопируй папку `chroma_data/` в корень проекта на новой машине — все документы и индексы сохранятся полностью. Переиндексировать ничего не нужно.
- **Резервная копия:** Достаточно заархивировать `chroma_data/` (например, `tar -czf chroma_backup.tar.gz chroma_data/`). Для восстановления — распакуй обратно.
- **⚠️ Важно:** `BM25`-индекс хранится **только в оперативной памяти** и не сохраняется на диск. После перезапуска сервера он автоматически пересобирается из данных ChromaDB — вмешательства пользователя не требуется.

```text
project_root/
└── chroma_data/       ← вся база здесь, просто скопируй эту папку
    ├── chroma.sqlite3
    └── ...
```

## 🚀 Установка и запуск

### 1. Клонирование и структура
Убедитесь, что структура проекта выглядит следующим образом:
```text
project_root/
├── .env
├── requirements.txt
├── backend/
│   ├── main.py
│   └── rag.py
└── frontend/
    ├── index.html
    ├── style.css
    └── script.js
```

### 2. Установка зависимостей
Рекомендуется использовать виртуальное окружение:
```bash
python -m venv venv
source venv/bin/activate  # Для Linux/Mac
# venv\Scripts\activate   # Для Windows

pip install -r requirements.txt
```
*(Пример `requirements.txt`: `fastapi uvicorn python-multipart chromadb openai langchain-openai langchain-core langgraph rank-bm25 pdfplumber pypdf python-docx python-dotenv tenacity`)*

### 3. Настройка окружения
Создайте файл `.env` в корне проекта и добавьте ваш API ключ:
```env
NVIDIA_API_KEY=nvapi-ваш-ключ-здесь
```

### 4. Запуск сервера
```bash
uvicorn backend.main:app --host 0.0.0.0 --port 8000 --reload
```
После запуска откройте в браузере: **http://localhost:8000**

## 📄 Рекомендации по качеству документов (Как получить лучшие ответы)

Качество ответов нейросети (RAG) **напрямую зависит от того, в каком виде загружены документы**. Нейросеть не видит картинки, цвета или сложное форматирование — она «читает» извлеченный голый текст.

### 🏆 Рейтинг форматов
1. 🥇 **Markdown (`.md`) и Plain Text (`.txt`) — Идеально.**  
   Текст максимально чистый. Алгоритм идеально бьет его на абзацы (чанки), нейросеть прекрасно понимает структуру (особенно если в Markdown используются заголовки `#`, `##` и списки).
2. 🥈 **Microsoft Word (`.docx`) — Хорошо.**  
   Система успешно извлекает абзацы и корректно читает простые таблицы, склеивая строки через `|`.
3. 🥉 **PDF (`.pdf`) — Удовлетворительно, но с нюансами.**  
   PDF — это визуальный формат, а не текстовый. Извлечение текста может сбиваться из-за переносов строк, колонтитулов или водяных знаков.

### 💡 Как подготовить идеальный документ для базы знаний:

* **Откажитесь от сканов (картинок в PDF):** Если PDF является набором отсканированных страниц без слоя распознанного текста (OCR), система выдаст ошибку, что текст не найден. *Решение: прогоните скан через FineReader или встроенный OCR в PDF-редакторе перед загрузкой.*
* **Явный контекст:** Местоимения ломают поиск. Если в документе написано просто *"Этот проект был сдан в 2023 году"*, нейросеть, найдя этот кусок текста, не поймет, о каком проекте речь. 
  * ❌ *Плохо:* «Договор расторгается при следующих условиях...»
  * ✅ *Хорошо:* «Договор аренды складов ООО "Ромашка" расторгается при следующих условиях...»
* **Упрощайте таблицы:** Сложные многоуровневые таблицы с объединенными ячейками (merged cells) в PDF или Word при извлечении превратятся в кашу слов. Старайтесь использовать простые плоские таблицы `Строка -> Столбец`.
* **Избегайте двухколоночной верстки:** В PDF текст в двух колонках часто парсится «зигзагом» (строка из левой колонки склеивается со строкой из правой). Если есть возможность, сохраняйте инструкции в одну колонку.
* **Списки вместо сплошного текста:** Нейросети гораздо легче обрабатывать нумерованные или маркированные списки, чем длинные запутанные абзацы на 20 строчек.

## 🛠 Под капотом (Архитектура поиска)

Запрос проходит через **LangGraph-граф** из трёх узлов:

1. **`retrieve`** — гибридный поиск:
   - **Векторная модель (`nv-embedqa-e5-v5`)** переводит вопрос в эмбеддинг и ищет тексты, похожие по **смыслу**.
   - **Алгоритм `BM25`** ищет точные текстовые совпадения (идеально для ИНН, ГОСТов, ФИО, номеров договоров). Тексты BM25-чанков подгружаются из ChromaDB точечно — только для тех, которых нет в векторных результатах.
   - **Reciprocal Rank Fusion (RRF)** объединяет результаты обоих поисков.

2. **`grade`** — фильтрация нерелевантных чанков по порогу косинусного сходства. BM25-чанки проходят без фильтрации — у них нет векторной дистанции.

3. **`generate`** — формирует контекст, вызывает LLM и возвращает ответ с метаданными:
   - `confidence` — откалиброванная уверенность (от 1% до 100%).
   - `retrieval_type` — тип поиска: `hybrid`, `vector_only` или `bm25_only`. При `bm25_only` уверенность фиксируется на 40%, так как векторная оценка недоступна.

Выбранные фрагменты отправляются в LLM с системным промптом, запрещающим придумывать факты (защита от галлюцинаций).

### 🔒 Надёжность и согласованность данных
- **Retry с backoff:** все запросы к NVIDIA NIM (эмбеддинги и LLM) автоматически повторяются до 4 раз при таймаутах или ошибке 429 (rate limit). Интервалы: 2 → 4 → 8 → 30 сек.
- **Параллельная индексация:** батчи эмбеддингов отправляются конкурентно через `asyncio.gather`. Глобальный `Semaphore(5)` ограничивает одновременные запросы к API, предотвращая rate limit при параллельной загрузке нескольких файлов.
- **Блокировка индексации:** каждый файл получает индивидуальный `asyncio.Lock`. Если один и тот же файл отправить на индексацию дважды — второй запрос подождёт, дубликатов не будет.
- **Однократная пересборка BM25:** при батчевой индексации нескольких файлов BM25 пересобирается один раз в конце, а не после каждого файла.
- **Реестр документов в памяти:** список файлов хранится в быстром словаре — без сканирования всей базы при каждом `/documents`. При перезапуске сервера восстанавливается из ChromaDB.
- **TTL для очереди загрузки:** файлы, загруженные через `/upload` но не проиндексированные в течение 1 часа, автоматически удаляются из памяти.
- **Прозрачная обрезка:** если документ порождает более 1000 чанков, лишние отбрасываются, а в лог пишется предупреждение с процентом потерянного контента.
